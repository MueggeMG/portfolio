01 — Motivation

Warum dieses Projekt?

Schach ist eines der komplexesten Spiele der Welt — gleichzeitig aber eines das sich perfekt eignet um echte Webentwicklung zu lernen. Es gibt klare Regeln, klare Zustände und ein greifbares Ergebnis.

Ich wollte kein Tutorial nachbauen. Ich wollte verstehen wie eine Schach-Engine kommuniziert, wie man Spiellogik von Darstellung trennt, und wie man eine externe Library wie Stockfish in einem modernen Projekt einsetzt.

"Ich entwickle digitalen Fortschritt — präzise, modern und mit einem Blick für Details."

Als Entwickler in Ausbildung war es mir wichtig ein Projekt zu bauen das über ein einfaches HTML/CSS Beispiel hinausgeht — mit echten Libraries, echtem Build-Tool und echtem Deployment.

02 — Tech Stack

Was ich genutzt habe
und warum.

Build Tool
Vite

Warum nicht plain HTML? Vite ermöglicht echte ES Module, Hot Reload und einen professionellen Build-Prozess. Genau das was in der echten Entwicklung genutzt wird.

Spiellogik
chess.js

Warum eine Library? Schachregeln selbst zu implementieren wäre Monate Arbeit. chess.js übernimmt Regelvalidierung, FEN-Notation und Spielzustand — ich konzentriere mich auf das Drumherum.

Brett & Figuren
Chessground

Warum Lichess' Library? Chessground ist das Brett hinter Lichess — einer der meistgespielten Schachplattformen der Welt. Bewährt, performant, erweiterbar.

KI Engine
Stockfish

Warum Stockfish? Die stärkste Open-Source Schach-Engine. Läuft als Web Worker — ein separater Thread damit die Seite nicht einfriert. Kommunikation über das UCI Protokoll.

Deployment
GitHub Pages

Warum GitHub Actions? Automatisches Deployment bei jedem Push. Kein manuelles Hochladen — echter CI/CD Workflow wie in professionellen Teams.

Design
Vanilla CSS

Warum kein Framework? CSS wirklich zu verstehen ist wichtiger als Tailwind zu benutzen. CSS Variables, Grid, Flexbox — alles von Hand, alles verstanden.

03 — Reflexion

Was ich mitgenommen habe.

Dieses Projekt hat mir gezeigt wie viele Schichten ein modernes Webprojekt hat — von der Spiellogik über die Engine-Kommunikation bis zum Deployment.

Der größte Lerneffekt war nicht der fertige Code, sondern der Weg dorthin. Jeden Fehler debuggen, jeden Begriff nachschlagen, jede Entscheidung bewusst treffen.

Das Projekt ist nicht perfekt — aber es ist ehrlich. Jede Zeile wurde verstanden bevor sie geschrieben wurde.

01
Web Worker & Multithreading Wie man rechenintensive Aufgaben in einen separaten Thread auslagert damit die UI flüssig bleibt.
02
UCI Protokoll Wie Schach-Engines über ein Textprotokoll kommunizieren — ein echtes Industriestandard-Protokoll.
03
Vite & Build Tools Wie ein modernes Build-Tool funktioniert und warum es in der professionellen Entwicklung unersetzlich ist.
04
GitHub Actions & CI/CD Wie automatisches Deployment funktioniert — Push, Build, Deploy in einem Workflow.
05
Separation of Concerns Spiellogik, Darstellung und Engine konsequent voneinander zu trennen.
